PowerShell script-writing gotchas (class-level, 2026-06-24): (1) UTF-8-saved .ps1 files with non-ASCII chars (`✓`, `→`, `—`, etc.) silently abort the parser mid-script. The error message points at the wrong character with "The term 'X' is not recognized" — X is parsed as a command, not a string. Fix: strip to ASCII (`✓` -> `[OK]`, `->`, `--`) or save with UTF-8 BOM (`Set-Content -Encoding UTF8` does NOT add BOM; use `[System.IO.File]::WriteAllText($path, $content, [System.Text.UTF8Encoding]::new($true))`). (2) Regex patterns inside double-quoted strings (`$config = $config -replace '^(\s*)port:\s*\d+', ...`) get parsed as commands — `\s`, `\d`, parens-with-content after are interpreted as command tokens. Fix: store the pattern in a variable first (`$pattern = '^(\s*)port:\s*\d+'`), OR write the regex work to a temp .py file and call python.exe to do it, OR use `[System.Text.RegularExpressions.Regex]::Replace` with a variable-stored pattern + `[System.Text.RegularExpressions.RegexOptions]::Multiline` (PowerShell's `-replace` operator doesn't support `(?m)` inline flag). (3) `cmd //c` from bash with a path containing spaces (`C:\Data\Hermes_0.17.0\...`) splits at the space and tries to run `C:\Data\Hermes` as a command. Fix: write a `.bat` wrapper that does the invocation, or use PowerShell's `&` operator which handles spaces natively.
